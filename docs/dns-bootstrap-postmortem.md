# Numa HAOS — DNS Bootstrap Latency: Complete Post-Mortem

## Status: RESOLVED (four root causes identified and fixed)

---

## Executive Summary

All non-cached DNS queries resolved by the Numa add-on showed `FORWARD` path at
`~3000ms` in the dashboard. Investigation across multiple sessions identified four
independent but interacting root causes:

| # | Root Cause | Effect | Fix |
|---|---|---|---|
| 1 | DoT upstreams on port 853 silently dropped by router | 3000ms per attempt × N upstreams | Replace DoT with DoH-only upstreams |
| 2 | Circular DNS bootstrap deadlock via hostname-based DoH upstreams | Cold queries pay one extra `timeout_ms`; DoH upstream reconnects slow | Add IP-based DoH upstream (`1.1.1.1`) |
| 3 | Auto-discovered `local.hass.io` forwarding rule → unreachable `169.254.169.253` | Every `*.local.hass.io` query times out at `timeout_ms` forever | Add explicit `[[forwarding]]` rule for `local.hass.io` |
| 4 | `172.30.32.3` (HA Supervisor dnsmasq) inaccessible from `host_network` container | All `getaddrinfo()` calls inside Numa fail with EAI_AGAIN; blocklist never downloads | Replace `172.30.32.3` with `127.0.0.1` in `resolv.conf` via run script |

After applying all four fixes: blocklist loads on first attempt, cached queries ≈ 0ms,
cold internet queries ≈ 50ms (after the first warm-up), `*.local.hass.io` queries ≈ 5ms.

### Important: resolv.conf is not the host's file

With `host_network: true`, Docker shares the **network namespace** (same IPs, same
ports) but generates a **separate** `/etc/resolv.conf` in the container's mount
namespace. The container always sees:

```
# Generated by Docker Engine.

nameserver 172.30.32.3
search local.hass.io
options timeout:10
```

`172.30.32.3` is the HA Supervisor's internal dnsmasq proxy. Docker generates this
file for every add-on container regardless of `host_network` setting. However, for
`host_network: true` containers, `172.30.32.3` is **not actually reachable** — see
Root Cause 4.

## Background: How Numa resolves DNS

The resolution pipeline in `src/ctx.rs` has three ordered branches:

```
1. Matching [[forwarding]] rule  → QueryPath::Forwarded  → "FORWARD" in dashboard
2. Recursive mode                → QueryPath::Recursive
3. Main [upstream] pool          → QueryPath::Upstream   → "UPSTREAM" in dashboard
```

`forward_with_failover_raw` (`src/forward.rs`) tries upstreams **sequentially**, not
in
 parallel. DoH upstreams always sort first (SRTT = 0, because `tracked_ip()` returns
`None` for `Upstream::Doh`). Hedging (`hedge_ms`) fires a second request to the **same**
upstream — it does not skip to the next one.

## Root Cause 1 — DoT upstreams on port 853

### Symptom

All queries showed `~3000ms` latency exactly matching `timeout_ms = 3000`.

### Original config

```toml
[upstream]
address = [
  "https://dns.quad9.net/dns-query",
  "tls://9.9.9.9#dns.quad9.net",
  "tls://149.112.112.112#dns.quad9.net",
]
timeout_ms = 3000
```

### Cause

Port 853 (DNS-over-TLS) is silently dropped by many home routers and ISPs.
`forward_dot_raw` opens a TCP connection to port 853; when the port is filtered the
connection hangs until `timeout_ms` expires. With two DoT upstreams and `timeout_ms =
3000`, a worst-case cold query takes up to 9000ms (though SRTT learning eventually
routes around failed upstreams after a few queries).

DoT also requires an IP address with a `#hostname` SNI suffix — a bare hostname like
`tls://dns.quad9.net` is rejected at parse time.

### Fix

Replace DoT with a second DoH upstream. Port 443 is never filtered.

```toml
address = [
  "https://dns.quad9.net/dns-query",
  "https://1.1.1.1/dns-query",
]
timeout_ms = 800
```

`timeout_ms = 800` is safe for healthy DoH servers (typically < 100ms round-trip) and
ensures a stalled upstream is bypassed quickly by the SRTT selector.

## Root Cause 2 — Circular DNS bootstrap deadlock

### The structural problem

When Numa's DoH upstream is hostname-based (e.g. `https://dns.quad9.net/dns-query`),
`reqwest` needs to resolve `dns.quad9.net` before it can open a TLS connection.
`reqwest` is compiled without the `hickory-dns` feature, so it calls libc
`getaddrinfo()`, which reads `/etc/resolv.conf`.

On
 a typical home network the router DHCP hands out the HA host IP as the LAN DNS
server. The container's `172.30.32.3` would (in theory) forward to the router, which
forwards to Numa — creating a circular dependency. In practice, `172.30.32.3` is also
unreachable from `host_network` containers (Root Cause 4), making the failure even
more fundamental.

### Why LAN client queries still work

LAN clients send UDP queries **directly to Numa's socket**. Numa owns the entire
pipeline end-to-end: even when DoH upstreams fail (because `getaddrinfo` loops or
times out), Numa eventually reaches its plain UDP fallback upstreams (`9.9.9.9`,
`149.112.112.112`) which use `UdpSocket::send_to` — **no `getaddrinfo`, no
intermediary**. The answer arrives late but Numa still sends it to the original LAN
client.

Internal `getaddrinfo()` calls (blocklist downloader, DoH reconnections) go through
`172.30.32.3`, which is unreachable — see Root Cause 4.

### Timeline — cold query with two hostname-based DoH upstreams

```
t=0ms     getaddrinfo("cdn.jsdelivr.net") → 172.30.32.3  [outer timer T starts]
            → 172.30.32.3 does not respond (Root Cause 4)
            → getaddrinfo() waits for timeout:10 (10 seconds)

t=10000ms getaddrinfo → EAI_AGAIN  ❌  (blocklist download fails, Attempt 1)
```

Even when Root Cause 4 is fixed (`172.30.32.3` replaced with `127.0.0.1`), a
hostname-based primary DoH upstream (`dns.quad9.net`) requires `getaddrinfo`, which
now goes to Numa itself. If Numa's first query is to resolve its own upstream, the
first cold query pays one extra `timeout_ms`:

```
t=0ms     DoH[0] "https://dns.quad9.net/dns-query"
            reqwest → getaddrinfo("dns.quad9.net") → 127.0.0.1:53 → Numa
            Numa: forward_with_failover_raw for "dns.quad9.net"
            → DoH[0] "dns.quad9.net" — needs to resolve itself → timeout_ms
t=800ms   DoH[0] fails

t=800ms   DoH[1] "https://1.1.1.1/dns-query" — IP literal, no getaddrinfo
            → direct TCP:443 → Cloudflare answers "dns.quad9.net" A record
            → getaddrinfo("dns.quad9.net") returns the IP
            → reqwest connects to dns.quad9.net → query answered
t=850ms   ✅
```

### PR #125 context

PR #125 (Numa v0.14.1) adds 3-retry backoff (2s / 10s / 30s) to the blocklist
downloader. Its commit message describes the fix as: *"by the second attempt the
upstream DoH connection is warm."* This is correct for the intended scenario (Numa's
`recv_from` loop not yet ready at startup). For the structural case, what saves later
attempts is **DNS cache side-population**: while processing the inner query, Numa
eventually resolves `dns.quad9.net` via the UDP fallback and caches the result. Later
attempts benefit from the cache hit.

### Fix — IP-based primary DoH upstream

When the URL host is an IP literal (`https://1.1.1.1/dns-query`), `reqwest` skips
`getaddrinfo()` entirely and opens the TLS connection directly. Cloudflare issues a
certificate valid for `1.1.1.1`, so TLS handshake succeeds without any DNS lookup.

With the IP-based upstream as **primary** (listed first), the blocklist downloader
and all internal HTTP clients connect to `1.1.1.1` directly, bypassing the circular
dependency entirely:

```toml
address = [
  "https://1.1.1.1/dns-query",             # primary — IP literal, no getaddrinfo
  "https://dns.quad9.net/dns-query",        # secondary — DNSSEC, no-log
]
```

### The fix that `reqwest` itself supports but Numa does not yet use

`build_odoh_client()` in `src/forward.rs` already uses `.resolve()` overrides on the
reqwest builder to bypass `getaddrinfo()` for ODoH relay/target hostnames:

```rust
pub fn build_odoh_client(odoh: &crate::config::OdohUpstream) -> reqwest::Client {
    let mut builder = https_client_builder(1);
    if let Some(addr) = odoh.relay_bootstrap {
        builder = builder.resolve(&odoh.relay_host, addr);  // ← bypasses getaddrinfo
    }
    ...
}
```

The same pattern applied to `build_https_client()` for regular DoH would eliminate
the circular dependency entirely, even with hostname-based URLs. This is the correct
upstream fix (see upstream issue filed against `razvandimescu/numa`).

## Root Cause 3 — Auto-discovered `local.hass.io` forwarding rule

### Discovery

`discover_linux()` in `src/system_dns.rs` reads `/etc/resolv.conf` and creates
forwarding rules from every `search` / `domain` line:

```rust
let forwarding_rules = if search_domains.is_empty() {
    Vec::new()
} else {
    let forwarder = resolvectl_dns_server()
        .unwrap_or_else(|| CLOUD_VPC_RESOLVER.to_string());
    // CLOUD_VPC_RESOLVER = "169.254.169.253"  ← AWS/GCP link-local address
    search_domains.iter().filter_map(|domain| make_rule(domain, &forwarder)).collect()
};
```

`resolvectl_dns_server()` queries `systemd-resolved` via D-Bus. Inside a Docker
container `resolvectl` fails (no D-Bus access to host `systemd-resolved`) → returns
`None` → forwarder falls back to `169.254.169.253`.

The container's resolv.conf has `search local.hass.io`, so Numa auto-creates:

```
ForwardingRule { suffix: "local.hass.io", upstream: Udp(169.254.169.253:53) }
```

`169.254.169.253` is an AWS/GCP link-local VPC resolver, **unreachable in any home
network**. Every query for `*.local.hass.io` (HA add-on ingress addresses, internal
service hostnames) hits this rule, waits the full `timeout_ms`, and returns SERVFAIL.

This is the source of the "FORWARD path at exactly `timeout_ms`" pattern seen in the
dashboard: not all forwarded queries were for general internet hostnames — the log was
dominated by HA's frequent internal `*.local.hass.io` lookups.

### Why this affects every HA OS installation using this add-on

`search local.hass.io` is injected by Docker/HA Supervisor into the container's
`/etc/resolv.conf` unconditionally, regardless of `host_network` setting. Any Numa
instance running as an HA add-on will trigger this code path.

### Fix — explicit forwarding rule overrides the auto-discovered one

An explicit `[[forwarding]]` rule in `numa.toml` is merged with — and takes priority
over — the auto-discovered rules via `merge_forwarding_rules()`. `172.30.32.3` is
the Supervisor's dnsmasq, which is authoritative for all `*.local.hass.io` names and
responds in < 5ms. Crucially, this forwarding rule is evaluated **inside Numa** (not
via `getaddrinfo`), so it routes correctly to `172.30.32.3` even though `172.30.32.3`
is unreachable for the host's own `getaddrinfo()` calls (Root Cause 4):

```toml
# Overrides the broken auto-discovered rule (169.254.169.253, unreachable).
# 172.30.32.3 is the HA Supervisor's dnsmasq — authoritative for all
# *.local.hass.io names (add-on ingress addresses, internal service hostnames).
[[forwarding]]
suffix   = "local.hass.io"
upstream = "172.30.32.3"
```

This rule is included **active and uncommented** in `numa.toml.default` because it is
universally correct for every HA OS installation.

## Root Cause 4 — `172.30.32.3` inaccessible from `host_network` container

### Discovery

After applying Root Cause 2's fix (`1.1.1.1` as IP-based primary upstream), the
blocklist download was still failing on all four attempts with identical 10-second
timeouts:

```
[WARN  numa::blocklist] attempt 1/4 failed: dns error: failed to lookup address
  information: Try again — retrying in 2s
[WARN  numa::blocklist] attempt 2/4 failed: ... — retrying in 10s
[WARN  numa::blocklist] attempt 3/4 failed: ... — retrying in 30s
[WARN  numa::blocklist] attempt 4/4 failed: ... — giving up
```

Each attempt took ~10 seconds (2 × 5s = glibc default `timeout:5` × `attempts:2`
against the single nameserver `172.30.32.3`). This was consistent across all four
attempts, including attempts 2–4 when Numa was definitely running and resolving LAN
client queries in 10–70ms.

Direct DNS test confirmed the fundamental problem:

```
docker exec addon_ee660d3e_numa nslookup cdn.jsdelivr.net 172.30.32.3
nslookup: write to '172.30.32.3': Connection refused
;; connection timed out; no servers could be reached
```

`172.30.32.3` actively refuses TCP connections and silently drops UDP queries.

Supporting diagnostics:

```
ha dns info
  locals: ["dns://192.168.1.1"]   # Supervisor uses router for general DNS
  servers: []                      # no custom DNS add-on configured in Supervisor

iptables -t nat -L OUTPUT -n | grep 53   # empty — no OUTPUT DNAT redirect
iptables -L OUTPUT -n | grep dpt:53      # empty — no filter rules on DNS
```

### Root cause

The HA Supervisor's dnsmasq (`172.30.32.3`) operates on the **Docker bridge network**
(`172.30.32.0/23`). It is designed to serve DNS exclusively for containers that have
a network interface on that bridge.

Numa uses `host_network: true`, which places the container in the **host's network
namespace**. The host shares the physical and virtual interfaces of the machine but is
**not attached to the Docker bridge as a client**. DNS queries from the Numa container
arrive at `172.30.32.3` sourced from the host's LAN IP (e.g. `192.168.1.X`), which
dnsmasq rejects (TCP RST) or silently ignores (UDP, no ICMP port unreachable returned).

Docker still generates `/etc/resolv.conf` with `nameserver 172.30.32.3` for every
add-on container regardless of `host_network` setting. The Supervisor provides no
mechanism to override this. The result: every `getaddrinfo()` call inside the Numa
process waits for `options timeout:10` (10 seconds) and returns `EAI_AGAIN`.

### Why other add-ons are not affected

**AdGuard Home** (also `host_network: true`, also a DNS server) does not exhibit this
problem because it is written in Go. Go's `net` package has a built-in pure-Go DNS
resolver that reads `resolv.conf` but implements its own UDP/TCP DNS client, independent
of libc. More importantly, AdGuard Home routes its own HTTP requests (blocklist
downloads, update checks) through its **internal DNS server** rather than through the
system resolver. It never calls `getaddrinfo()` for those operations.

Numa uses `reqwest` (Rust HTTP client) which calls libc `getaddrinfo()` by default
for hostname resolution. This is the structural difference.

### No structural solution in HA

The HA Supervisor `config.yaml` schema exposes 55 documented configuration keys.
None of them configure the container's DNS resolver. There is no `dns:` field or
equivalent. The only DNS-related mechanism is `host_network: true` itself (which
grants access to host port 53 for *serving* DNS) and `ports` (which exposes ports).

A `dns:` field analogous to Docker Compose's `dns:` key would be the structural fix.
Until that exists in the Supervisor, the run script is the correct place to fix
`resolv.conf`.

### Fix — replace `172.30.32.3` with `127.0.0.1` in the run script

With `host_network: true` and `bind_addr = "0.0.0.0:53"` in `numa.toml`, Numa listens
on the loopback interface. `127.0.0.1:53` **is Numa itself** — always reachable, no
network hops, no intermediary.

The run script replaces the unreachable nameserver before `exec`-ing Numa:

```bash
if ! grep -q '^nameserver 127\.0\.0\.1' /etc/resolv.conf 2>/dev/null; then
    sed 's/^nameserver 172\.30\.32\.3$/nameserver 127.0.0.1/' /etc/resolv.conf \
        > /tmp/resolv.conf.numa \
        && cat /tmp/resolv.conf.numa > /etc/resolv.conf
fi
```

After the fix, the DNS chain for all internal `getaddrinfo()` calls is:

```
getaddrinfo("cdn.jsdelivr.net")
  → 127.0.0.1:53  (Numa itself, loopback)
  → https://1.1.1.1/dns-query  (IP literal — no further getaddrinfo)
  → Cloudflare answers: 104.16.85.20
  → getaddrinfo returns in ~100ms  ✅
```

**Startup timing is safe.** Numa binds port 53 during tokio runtime initialization
(< 10ms after `exec`). The blocklist downloader's first `getaddrinfo()` call occurs
after HTTP client setup (~100–150ms into startup). There is no race.

The `cat >` (in-place write) pattern preserves the original inode because Docker
bind-mounts `/etc/resolv.conf`; replacing the inode with `mv` would silently break
the bind-mount for the container lifetime.

**Note on `172.30.32.3` in forwarding rules:** Numa's internal DNS forwarding for
`local.hass.io → 172.30.32.3` (Root Cause 3 fix) is not affected. That rule is
evaluated inside Numa's own UDP forwarding stack (`forward_udp_raw`), which binds a
source socket on the container's (host's) network. Numa CAN reach `172.30.32.3` as
a forwarding target; only `getaddrinfo()` from the host's own processes cannot.

## Upstream fix — PR #126 (merged 2026-04-21)

Issue #122 was filed against `razvandimescu/numa` describing:

1. The structural `2T > T` deadlock for hostname-based DoH upstreams in circular-DNS
   environments (router DHCP → HA host as DNS server).
2. That PR #125's retry mechanism partially mitigates the issue via DNS cache
   side-population, but does not fix the root cause.
3. The proposed fix: apply `.resolve(hostname, bootstrap_ip)` on the reqwest builder
   for regular DoH clients, mirroring the pattern already used in `build_odoh_client()`.
4. In `host_network: true` add-ons the system nameserver (`172.30.32.3`) is
   structurally unreachable, making all `getaddrinfo()` calls fail unconditionally —
   a second, independent failure mode for the same class of bug.

**PR #126 (`fix/self-resolver-loop`) was merged into `main` on 2026-04-21.**

### What PR #126 implements

`NumaResolver` — a new `src/bootstrap_resolver.rs` (236 lines) implementing
`reqwest::dns::Resolve` with two resolution lanes:

- **Override lane**: per-host `HashMap<String, Vec<IpAddr>>` fed from ODoH
  `relay_ip`/`target_ip` config — zero DNS queries for those hosts.
- **Bootstrap lane**: parallel A + AAAA queries via raw UDP to IP-literal bootstrap
  servers, with automatic TCP fallback if UDP is blocked. Bootstrap IPs are drawn
  from `upstream.fallback` (IP-literals only); if `fallback` is absent or contains
  only hostnames, falls back to hardcoded `[9.9.9.9:53, 1.1.1.1:53]`.

`NumaResolver` is constructed once at startup, shared as `Arc<NumaResolver>`, and
wired into **every** Numa-originated HTTPS client:
- All DoH upstream clients (`parse_upstream_list`)
- ODoH relay/target client (replaces the old per-host `.resolve()` overrides)
- Blocklist downloader (`download_blocklists` gains `resolver` parameter)
- Auto-mode DoH fallback

`doh_keepalive_loop` now fires **immediately** at boot (the `.tick().await` skip is
removed), surfacing bootstrap failures within ~100ms instead of waiting for the first
query.

A Docker-based regression test (`tests/docker/self-resolver-loop.sh`) reproduces
exactly our failure scenario: `/etc/resolv.conf → 127.0.0.1`, hostname-based DoH
upstream, blocklist URL. **Before fix**: 0 domains loaded, 3072ms SERVFAIL.
**After fix**: 397k domains loaded, 118ms NOERROR.

### Impact on this add-on (target: Numa v0.14.2)

| Current workaround (v0.14.1) | Status after v0.14.2 |
|---|---|
| `resolv.conf` injection: replace `172.30.32.3` → `127.0.0.1` in run script | **Remove** — `NumaResolver` bypasses `resolv.conf` entirely |
| `https://1.1.1.1/dns-query` as IP-literal primary upstream | Optional — hostname-based DoH now bootstraps correctly; keep for resilience |
| `fallback = ["9.9.9.9", "149.112.112.112"]` | **Keep** — now serves double duty as `NumaResolver` bootstrap IPs |
| `warm` list for startup cache pre-population | Less critical — `doh_keepalive_loop` fires immediately at boot |

**`/etc/resolv.conf rw`** in `apparmor.txt` can be downgraded to `r` once the run
script injection is removed.

The `[[forwarding]] local.hass.io → 172.30.32.3` rule (Root Cause 3) is unaffected
— it is evaluated inside Numa's own UDP forwarding stack, not via `getaddrinfo`.
